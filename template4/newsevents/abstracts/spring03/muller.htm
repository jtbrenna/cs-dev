Chestnut Hill, Massachusetts

Abstract

 

A well-known and embarrassing fact of life in computer science is that virtually every non-trivial software system wrought by the hands of humans is incorrect. That is, there are almost always circumstances, usually very many, where the software system doesn’t perform the way the software client wants or expects it to. Techniques for increasing the likelihood that software meets the consumer’s expectations have been studied extensively over the years. An apparently tractable sub-goal of this work has been to attempt to prove the correctness of the programming language implementation (i.e., the compiler). This would indeed justify some increased confidence on the part of the consumer. Unfortunately, after forty years of concerted effort, even this sub-goal remains elusive, especially for production-quality compilers.

 

Recently the notion of a certifying compiler has taken root. A certifying compiler emits object code together with an efficiently checkable (and reliable) certificate that the program satisfies a given property. For example, it may certify that the object code does not violate some constraints on memory references. Like all real compilers, certifying compilers are almost certainly incorrect. However, the reliability of the certified property of the object code can be mechanically checked before the code is executed.

 

In this talk, I will introduce a system that is designed to support inter-procedural register allocation and the representation of tuples and variants directly in the register file.  I will also present an algorithm that generates constraints for assigning annotations. The algorithm is sound with respect to the type system.  This soundness result is stronger than the usual soundness results for typed assembley languages.

 

This is joint work with Torben Amotft of Kansas State University.
